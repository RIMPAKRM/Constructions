# Система авторизации шкафа (Auth Cabinet Implementation)

## Статус: ПОЛНОСТЬЮ РЕАЛИЗОВАНО И ПРОТЕСТИРОВАНО ✓

### Описание системы
Система авторизации управляет доступом к дверям и люкам в пределах 40-блочного радиуса от шкафа авторизации.

---

## Реализованные функции

### 1. **Сообщения авторизации**
- ✓ Сообщения отправляются **только взаимодействующему игроку** (не broadcast)
- ✓ Используется `player.displayClientMessage()` вместо `broadcastSystemMessage()`
- ✓ Сообщения имеют цветовое кодирование:
  - `§a` (зелёный) - успешная авторизация
  - `§6` (золотой) - список авторизованных
  - `§c` (красный) - ошибки/запреты

### 2. **Управление авторизованными игроками**
- **ПКМ (правый клик) на шкафе авторизации:**
  - Если игрок **НЕ авторизирован**: добавляет его в список авторизованных
  - Если игрок **уже авторизирован**: показывает полный список авторизованных + владельца
  
- **ЛКМ (левый клик) на шкафе авторизации:**
  - Только владелец может очищать список
  - При очистке удаляются **ВСЕ авторизованные** (включая владельца из списка)
  - Владелец при этом сохраняет доступ через механизм `isPlayerAuthorized()`

### 3. **Проверка авторизации для дверей и люков**
- ✓ При попытке открыть дверь или люк проверяется авторизация в ближайшем шкафе
- ✓ Если нет ближайшего шкафа — доступ **разрешён** (свободное строительство)
- ✓ Если есть шкаф — только авторизованные игроки могут открыть
- ✓ **Владелец всегда имеет доступ** (через `isPlayerAuthorized()`)
- ✓ Неавторизированным показывается сообщение: `§c[Доступ запрещён] Вы не авторизированы в шкафе авторизации!`

### 4. **Визуал шкафа**
- Модель: **Lectern** (лектерн)
- Блок: `Blocks.LECTERN` в игре
- Item Model: `auth_cabinet_item.json` с текстурой `minecraft:block/lectern`

### 5. **Ограничения и правила**
- Только **1 шкаф авторизации** может находиться в радиусе 40 блоков
- Шкаф может быть размещен только авторизированными игроками
- При разрушении шкафа структура и все двери/люки остаются (но теряют управление)
- Радиус авторизации: **40 блоков** от центра шкафа

## Логика авторизации (подробно)

### Метод `isPlayerAuthorized(UUID playerId)`
```
Возвращает true если:
  - playerId находится в списке авторизованных ИЛИ
  - playerId является владельцем шкафа
```
**Используется для:** открытие дверей/люков, размещение структур в зоне

### Метод `isPlayerInAuthList(UUID playerId)`
```
Возвращает true если:
  - playerId находится в списке авторизованных (ТОЛЬКО)
```
**Используется для:** отображение списка при ПКМ на шкафе

### Метод `clearAllAuthorizedPlayers()`
```
Очищает весь список авторизованных (включая владельца из списка)
Владелец сохраняет доступ через isPlayerAuthorized()
```
**Используется для:** ЛКМ на шкафе авторизации

## Тестирование в игре

### Сценарий 1: Авторизация игрока
1. Игрок нажимает ПКМ на шкафе
2. Игрок авторизируется (добавляется в список)
3. Игрок видит сообщение **только он один** (не другие игроки)

### Сценарий 2: Просмотр списка авторизованных
1. Авторизированный игрок нажимает ПКМ на шкафе снова
2. Видит список всех авторизованных + владельца
3. Сообщение видит **только этот игрок**

### Сценарий 3: Очистка авторизованных
1. Владелец нажимает ЛКМ на шкафе
2. Все авторизованные удаляются из списка
3. Владелец видит подтверждение: `§c[Шкаф авторизации] Все авторизованные игроки удалены (включая вас).`

### Сценарий 4: Открытие двери
1. Авторизированный игрок подходит к двери рядом (в пределах 40 блоков от шкафа)
2. Нажимает ПКМ на двери
3. Дверь открывается

### Сценарий 5: Блокировка неавторизированного
1. Неавторизированный игрок подходит к двери
2. Нажимает ПКМ на двери
3. Видит сообщение: `§c[Доступ запрещён] Вы не авторизированы в шкафе авторизации!`
4. Событие отменяется — дверь не открывается
